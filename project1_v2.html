개발 배경 Background

SAP 표준 전표 전기 트랜잭션은 현업 부서의 대량 일괄 결재 및 검증 프로세스를 효율적으로 지원하지 못했습니다. 보안 강화 및 회계 감사 프로세스 준수를 위해 전표 확정 및 원장 반영 전에 다단계 검증 처리가 가능한 솔루션이 필요했습니다.

이에 따라 입금 정보 처리를 위한 ZNS_FI_INPAY 프로그램과 출금 전표 처리를 위한 ZNS_FI_OTPAY CBO 프로그램을 설계했습니다. 이 솔루션을 통해 실무 담당자가 업로드한 원시 데이터를 실시간으로 파싱하고 임시 테이블에 적재하여 정합성을 사전에 검증할 수 있는 환경을 마련했습니다.

The standard SAP payment transaction did not natively support the client's specific bulk approval and posting workflows. For security and audit compliance, the finance team required multi-level batch processing before documents could be committed to the ledger.

To solve this, I designed two core CBO programs: ZNS_FI_INPAY (for incoming payment batches) and ZNS_FI_OTPAY (for outgoing payment batches). These custom programs allowed operators to upload, stage, and pre-verify bulk transactions prior to final posting.

트러블슈팅 & 해결 방안 Troubleshooting & Solution

문제 상황: 1,000건 이상의 대량 전표를 일괄 처리하는 스트레스 테스트 중, 데이터 불일치(Data Consistency) 문제가 발생했습니다. 루프 도중 특정 데이터 오류로 인해 일부 항목만 전표 전기가 성공하고 나머지는 오류가 나면서, 임시 저장 테이블과 실제 회계 원장의 데이터가 불일치하는 심각한 원장 불일치 현상이 나타났습니다.

해결책: 일괄 처리 프로세스의 안정성을 극대화하기 위해 SAP 표준 전표 처리 함수인 BAPI_ACC_DOCUMENT_POST를 활용하고, 이를 트랜잭션 단위로 묶는 래퍼(Wrapper) 로직을 개발했습니다. BAPI 처리 후 결과 로그(Return Table)를 파싱하여 단 한 건의 에러('E')라도 감지되면 즉시 BAPI_TRANSACTION_ROLLBACK을 수행하고 전체 트랜잭션을 취소하도록 하였으며, 모든 항목이 유효할 때만 COMMIT WORK AND WAIT를 실행하여 전표의 데이터 무결성을 보장했습니다.

Challenge: During stress testing with large bulk data (1,000+ records), we encountered data consistency issues. Partial document posting occurred due to logical errors mid-execution, leaving some records posted while others rolled back, causing ledger imbalances.

Solution: I re-architected the batch logic to utilize the standard SAP function module BAPI_ACC_DOCUMENT_POST. I wrapped the posting loop in a transaction wrapper with strict COMMIT WORK AND WAIT and ROLLBACK WORK logic. If any single document posting in the batch failed validation, the entire database transaction rolled back, ensuring complete "all-or-nothing" data integrity.

zns_fi_post_wrapper.abap ABAP 
* Call posting BAPI for transaction item
CALL FUNCTION 'BAPI_ACC_DOCUMENT_POST'
  EXPORTING
    documentheader    = ls_header
  TABLES
    accountgl         = lt_accountgl
    currencyamount    = lt_currencyamount
    return            = lt_return.

* Read return log to check for errors
READ TABLE lt_return WITH KEY type = 'E' TRANSPORTING NO FIELDS.
IF sy-subrc = 0.
  CALL FUNCTION 'BAPI_TRANSACTION_ROLLBACK'.
  ls_log-status = 'E'.
  ls_log-message = 'Error during document post. Rolled back.'.
ELSE.
  CALL FUNCTION 'BAPI_TRANSACTION_COMMIT'
    EXPORTING
      wait = 'X'.
  ls_log-status = 'S'.
  ls_log-message = 'Document posted successfully.'.
ENDIF.

사용자 경험 (UX) User Experience (UX)

시스템 관리자와 현업 부서 사용자가 일괄 처리 결과를 편리하게 감사(Audit)할 수 있도록 상세 ALV 로그 그리드를 구축했습니다. 그리드 행에 신호등(Traffic Light) 및 색상 필터링 규칙을 적용하여 성공건은 녹색, 에러건은 적색으로 하이라이트하고, 에러 행을 더블클릭할 시 즉시 오류가 발생한 데이터 필드로 포커스가 이동하는 드릴다운 기능을 추가하여 유지보수 소요 시간을 크게 줄였습니다.

To ensure system administrators and finance users could audit execution logs seamlessly, I developed a detailed ALV success/error log grid. The ALV report dynamically color-coded rows (green for successful postings, red for errors) and allowed users to double-click on any error line to drill down directly to the offending data field, reducing troubleshooting times significantly.
